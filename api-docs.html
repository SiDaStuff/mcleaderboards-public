Public API Documentation

Access leaderboard data programmatically

Base URL

https://mcleaderboards.org/api

All endpoints are relative to this base URL.

Authentication

Most endpoints require authentication. Include the Bearer token in the Authorization header:

Authorization: Bearer YOUR_JWT_TOKEN

Get your token by logging into your account and using the authentication endpoints.

Endpoints

GET /health

Check API availability

Response: 
{
  "status": "ok",
  "timestamp": "2024-01-01T00:00:00.000Z"
}

GET /players

Get players with pagination support (optimized for infinite scroll)

Parameters:
  • gamemode (optional): Filter by specific gamemode (vanilla, uhc, pot, nethop, smp, sword, axe, mace, overall)
  • offset (optional): Starting position for pagination (default: 0)
  • limit (optional): Number of players to return (default: 25, max: 100)
Examples: 
GET /api/players?gamemode=vanilla&offset=0&limit=25
GET /api/players?offset=25&limit=25
GET /api/players?gamemode=pot
Response: 
{

      "username": "PlayerName",

Rate Limiting

  • 100 requests per minute for general endpoints
  • 5 requests per 15 minutes for authentication endpoints
  • 30 requests per minute for queue operations

Rate limits help ensure fair usage of the API and prevent abuse.

Caching

The API uses intelligent caching to improve performance:

  • Player data: Cached for 5 seconds
  • Dashboard stats: Cached for 5 seconds
  • Blacklist: Cached for 30 seconds

Cached responses include a "cached": true field in the response.

Pagination Best Practices

When implementing pagination with the /api/players endpoint:

  • Start with offset=0 and limit=25
  • Check the hasMore field to determine if more pages exist
  • Increment offset by limit for each subsequent request
  • Use the total field to display total player count
  • Implement infinite scroll by loading more when user scrolls near bottom

Example sequence:

1. GET /api/players?offset=0&limit=25
2. GET /api/players?offset=25&limit=25
3. GET /api/players?offset=50&limit=25
...

Error Handling

All errors follow this format:

{
  "error": true,
  "code": "ERROR_CODE",
  "message": "Human readable error message"
}

Common HTTP Status Codes:

  • 200 - Success
  • 400 - Bad Request (validation error)
  • 401 - Unauthorized (invalid/missing token)
  • 403 - Forbidden (insufficient permissions)
  • 404 - Not Found
  • 429 - Too Many Requests (rate limited)
  • 500 - Internal Server Error
